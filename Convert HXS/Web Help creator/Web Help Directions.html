Directions for Creating Web Help From HXS Files

    Web Help here refers to a viewer for help webpages. It's not clear why Microsoft uses the confusing terms "web help" and "online help" for help files that are not online, but since those terms have become commonly used, "web help" is used here to refer to documentation in the form of .HTM webpages.

    The Web Help scripts will unpack any number of HXS files, store the resulting .HTM files, and build an index/viewer in the form of an HTA named index.hta. The file index.hta, when opened, will present a window similar to an MSDN help viewer, without any need to actually install SDKs, Visual Studio, or anything else.

Note: The following directions require scripting drag-drop operation. On Windows Vista/7, if you are running with typical security restrictions, dropping files and folders onto .VBS files may not work. In that case you might have to edit one or more of the scripts.

Directions:

0) 7-Zip must be installed for these scripts to work. 7-Zip is used to unpack .HXS files. With 7-Zip installed, open Create WebHelp From HXS.vbs, scroll down to the "SETTINGS" section, and confirm that the path there matches the 7-Zip path on your system.

1) Create an empty folder for the Web Help, in a permanent location where you will keep the Web Help after it's created.

2) Collect all relevant .HXS files into a single folder. (A different folder from the new Web Help folder.)

3) Drop the folder full of HXS files onto the script named Create WebHelp From HXS.vbs. The script will then show an Inputbox, in which you should enter the full path of the newly-created Web Help folder.

   Expect the script to run between 10 and 30 minutes for a large Platform SDK collection of 100+ HXS files. When the script is finished it will show a message. At that point the Web Help folder will contain one folder full of .HTM files for each .HXS file processed. There will also be a file index.hta, which is the help viewer. And there will be a file named wordlist.dat, which is a list of topic titles. wordlist.dat should be left undisturbed. index.hta loads the list in wordlist.dat at startup in order to provide fast search functionality for the index. (A Web Help viewer created for the Server 2003 SDK had over 87,000 entries in the index. With so many entries scrolling alphabetically can be tedious.)

4) [Optional] Open the file Fix Webhelp Links.vbs in Notepad if you want to customize the help pages with CSS. A STYLE block can be added near the top of the script, which will be inserted in all .HTM files. (There is a sample STYLE block commented out.)

5) Drop the new Web Help folder onto the script Fix Webhelp Links.vbs. Expect that script to run for perhaps 3-15 minutes for a large Platform SDK Web Help project. A message will show when the script is finished.

    Fix Webhelp Links.vbs does several things. One of those is critical: HXS webpages often have corrupted links that use a nonsensical "<MSHelp>" HTML tag. If those links are not repaired then many links in the help pages will not work. Fix Webhelp Links.vbs also, optionally, does two other things: 1) It can add custom style, as explained above, allowing you to customize page colors, font properties, heading styles, etc. for the help pages in the Web Help viewer. 2) It can optionally remove javascript from the help pages. When Fix Webhelp Links.vbs runs you will be provided with a choice of whether to do that. Script removal is offered because the Microsofties tend to get carried away with superfluous script functionality that only gets in the way. Example: The help page for the IE Document object has closed lists of Properties, Functions and Events. You must click Properties to see the list expanded; then click again to fully open the list. If you then click to see the Functions list, the Properties list closes again! This irritating design -- using javascript to toggle the visibility of page content -- is quite common in MS help files. If you choose to remove script, that function will also remove style "display: none;" code, with the result that all page content will be visible and all now-you-see-it-now-you-don't nonsense will be removed. The IE Document object page will simply show a list of all properties, functions and events.

6) The Web Help viewer is now ready for use. Leave the Web Help folder where it is. (The index.hta file has hard-coded file paths to the help webpages. If you move the folder the help viewer will not work.) Create a shortcut to index.hta on the Desktop, in the Start Menu, etc., and use that to view the help. When the viewer is opened it will take a moment to load the index and the wordlist.dat file. Once loaded, you can search the index and click any item in the index to view that page. (There is no option to show a "Contents" view, and if you need to search all pages for a specific string you will need to use something like Agent Ransack. The built-in search only provides the ability to auto-scroll to a topic.)

Brief Explanation: There is more info. provided elswhere about HXS files. Basically, CHM (HTML Help 1) and HXS (HTML Help 2) are almost the same thing. They are both simply websites compressed into a single file, usually with an index. The .HTM files that comprise the help topics can be extracted. What the Web Help viewer scripts do is to extract all of the webpages from HXS files, create a viewer for those webpages. They also perform necessary repairs on the code in the .HTM files. And they generate a master index, using the topic names from each of the .HTM files. The final result is similar to an MSDN help viewer, with all help files joined into one single viewer, with a single, master index.

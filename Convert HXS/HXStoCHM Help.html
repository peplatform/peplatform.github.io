Introduction

Update note: See update history below. If you converted HXS files and empty CHMs were produced, this update should fix the problem.

HXStoCHM is a set of files for converting HXS to CHM. CHM files are Microsoft HTML Help format. HXS is the "Help 2" format.

Microsoft is increasingly distributing their documentation as one giant platform SDK. The help in that SDK is HXS, designed for use with VS.Net. Microsoft has not made a stand-alone HXS viewer program available. Worse, the SDK installations are gigantic, bloated affairs that are designed to work only in a very limited way. For instance, the Windows 7 SDK is designed to install ONLY on Windows 7! Anyone who prefers to develop on a different version of Windows is out of luck.

The point of HXStoCHM is to provide a way for people using C++, VB, Delphi, etc. -- not in VS.Net -- to use updated API docs, and to access any other HXS files they might come across, without having to install a lot of bloat they don't want just to get a Help2 reader.

This updated version has 3 options:

Multiple HXS to CHM conversion\ConvertToCHM2.vbs -- Drop a folder containing 1 or more HXS files onto this script and it creates CHM versions.

HTA\HXSConv.hta -- A fullscale GUI program written as a VBScript-powered HTA webpage. (See the picture hxsc.gif.) This option provides all the functionality of the first script in a convenient interface. No script edting required.

Web Help creator\Create WebHelp From HXS.vbs -- Drop a folder containing 1 or more HXS files onto this script and it creates an HTA reader utility with a common index. The HXS files are unpacked to create a folder tree of HTML topic pages. They are not converted to CHM. (The Web Help creator scripts have been updated twice, most recently in January, 2012. With the current version, pages in the help can be customized with custom STYLE code, the scripts can remove all annoying javascript from help pages, and the viewer window has been improved, with fast index search functionality.)

What about getting the HXS files?
Microsoft makes several SDKs available. The Server 2003 SDK, for example, is a good one for people who want up-to-date docs for XP but are not interested in .Net docs.

Typically the SDK downloads are in self-executing CABs. The XtoCAB utility included here can convert a CAB EXE to a CAB. The utilities at http://www.jsware.net/jsware/msicode.php5 can be used to unpack any MSIs involved with SDK packaging. With those two tools it should be no problem to get the HXS files from an SDK without having to "install" the whole thing.

In the case of the Windows 7 SDK it is available as a DVD ISO file. (Often named with the extension .IMG.) The file can be unpacked by 7-Zip. (It seems to be nothing more than all the SDK files pasted together, anyway. Unpacked it is the same size as the original ISO file.) Once the ISO is unpacked, one can use the utilities at http://www.jsware.net/jsware/msicode.php5 to find the HXS files by opening the various MSI files. (The HXS files are stored in CAB files, which are accessible, but the file names a mangled and obscured by the Windows Installer MSI creation. An MSI unpacker tool is not required, but it will make the job much easier, unpacking the CABs into a folder tree and fixing the file names as it goes.

Requirements

1) 7-Zip is required to unpack the HXS files. (An HXS is a compressed format, like a ZIP file.)
Download 7-Zip here: http://www.7-zip.org/

2) Microsoft HTML Help Workshop is required to compile the CHM file.
Download Help Workshop here: Download HTML Help Workshop

3) Some minor editing of the scripts is required. Open ConvertToCHM2.vbs in Notepad or a script editor and configure the Settings near the top. There are 2 required settings and several optional settings, all very simple. They are explained in the script comments.

With those 3 requirements taken care of, the script is ready to use. Just drop a folder onto it.

Create WebHelp From HXS.vbs has similar settings to adjust.

Fix Webhelp Links.vbs, the script used after Create WebHelp From HXS.vbs when creating a web help setup, has only one optional edit: If you want to add custom CSS to all webpages it needs to be added near the top of the script. The script comes with a sample STYLE block that has been commented out. Adding any amount of CSS code there will add that code as a STYLE block at the top of all webpages processed. That allows you to add backgrounds, change fonts, customize headings, etc.

HXSConv.hta does not require any special editing. The settings are handled through the interface.

What is HXS and how does HXStoCHM work?

HTML Help files (CHM) are a compiled file that packs in numerous HTML files with an index and a table of contents (TOC). The index file and TOC file are basically HTML files themselves.

HXS files are very similar. The main difference is that the format of the TOC and index has been changed to an XML-ish design.

Microsoft started with HLP files, which were based on Word docs. Then the Internet became fashionable and Microsoft jumped on the bandwagon, creating "HTML Help" that looks like webpages. (In case you don't get the point there's a spinning Earth cartoon in most of them, to tell you that you're reading an up-to-date, "Internetty" help file.)

Help 2 (HXS files) reperesents a second major jumping onto a bandwagon. XML became all the rage and Microsoft acted in character: They made all sorts of pointless changes to HTML Help, "updating" the HTML TOC and index files syntax to XML.

Aside from those minor, frivolous changes, Help1 and Help2 seem to be almost the same thing. Both CHM and HXS files can be opened by 7-Zip. Both contain a TOC file, usually an index, a few other odds and ends, and webpage files that represent the help topics. Both use a similar project file, except that Help1 uses an INI format in the HHP file while Help2 uses XML format in its HXC file.

HXStoCHM works by unpacking the HXS file, then converting the TOC and index to CHM versions, then creating a CHM-type project file, then calling Help Workshop to compile the whole thing.

The operation is mostly just text parsing. It's a bit tricky because the format of the HXS TOC and index can vary. But in essence HXS varies little from CHM. The relevant file types are as follows:

File Help2 Help1
Help file HXS CHM
TOC HXT HHC
Index HXK HHK
External index HXI CHI
Project file HXC HHP
File list HXF part of HHP


A valid HXK(index) file is not always present. When the HXS has no index an index is created by using the topics in the TOC. In other words, each topic title becomes an alphabetical entry in the index.

The HXC file is used here only to get the title for use in the CHM file name and help window title bar. (Though a title is often not present.)

The HXF file is not used. It corresponds to the file list in an HHP file. The script instead just enumerates the subfolders and files unpacked from the HXS, adding any "web files" to the list. The critical point is that the relative paths of these files must be correct for the whole thing to work. Having worked out the relative paths of all relevant files, the script adds that list to the HHP file.

So the overall process goes like so:

* Call 7-zip to extract the HXS contents.

* Rummage around in the contents to find the HXT file.

* Create an HHC(TOC) from the HXT(TOC).

* If there is a valid HXK(index) present then use it to create an HHK(index) file. Otherwise, create an HHK from the HHC.

* Get a list of files, with their relative paths. The list includes all web files (htm. jpg, etc.) that are in the folder with the HXT, or in subfolders with names that do not begin with $. (This is one of the ways in which HXS files can vary. The content files may be loose in the HXS; they may be in a subfolder; there might be a dozen subfolders. One can't assume. So the script uses a recursive file enumeration function. One could parse the HXF file to get a file list, but enumerating existing files is fairly easy and avoids the possibility that HXF files may be as undependable in their format as the other Help2 files.)

* If a meaningful file name is desired then check the HXC file to see if a title has been assigned to the project. Often there is no title, but sometimes there's a name that's more informative than the HXS file name.

* Create an HHP project file that lists the necessary values for the HTML Help compiler and also lists all of the relative file paths.

* Finally, call the HTML Help Workshop compiler to create a CHM file.

The script waits for 7-Zip, then waits again for the Help compiler to do its work. It uses a separate script to call 7-Zip in order to make sure that 7-Zip exits after each extraction. That design allows for numerous HXS files to be processed smoothly, one after the other. (If the main script calls 7-Zip it will remian loaded until the script quits, resulting in numerous running instances of 7-Zip.)

MSDN integration

If CHM files are created with CHIs they can be integrated into earlier, CHM-based versions of MSDN, the help library reader that comes with Visual Studio. At this link are directions and a utility for integrating any CHM into MSDN:

http://www.codeguru.com/cpp/w-p/help/msdn/article.php/c6507/

Note: You may want to edit the index by hand in some cases. The translation from HXK to HHK files is not perfect. In some cases it creates redundant entries. That's not a problem in a single CHM, but it may be undesirable in the MSDN index. It's advisable to look over the index of any CHM you intend to integrate before proceeding.

Technical notes

Index files

Some HXS files have a number of index files inside with names using the protocol of *K.hxk, *A.hxk, *F.hxk, etc. These are optional files that are typically empty, apparently generated automatically by the program used to create the HXS file. When you open an HXS file there will often be 1 HXK file that is much bigger than the rest. That is a usable index file. (In some cases there's a main index file but it has no valid content.)

The Search tab

HXKtoCHM adds a Search tab to the help file. It does that by adding the setting to the HHP project file. To see how that works, open a CHM project in Help Workshop. Open the OPTIONS settings by clicking the top left button in Help Workshop. Then click the Compile tab and check the box marked "Compile full text search information". That creates an entry in the project settings: "Full-text search=Yes". Those settings are in the HHP file, which the compiler uses to create a CHM.

Creating CHI files

There is an option in HXStoCHM settings to create a CHI file. That's useful if, for instance, you want to add the CHM to MSDN. Strangely, the Microsoft HTML Help Workshop treats that option as a "hidden" setting! It is not a visible option. Instead a line has to be added to the HHP file options by hand: Create CHI file=Yes. Or rather, there is an option (a checkbox) to create a CHI, but it doesn't work! HXStoCHM2 provides the option in the script settings and acts accordingly when it writes the HHP file.

Fixes in Jauary 2018 update
Errors were found in the Windows 7 SDK. Some HXS files have a faulty HXT file that resulted in an empty CHM conversion. Two examples are script56 and automation. It appears that Microsoft have redesigned the HXS format and used the new design in an arbitrary manner. This update accomodates that problem.

   Please feel free to send an email if you find any problems with conversions. For those who are curious, open the script56 and the shell HXS files in 7-Zip. The former is the new style. The latter is the old style. The HXT file in the old style lists a title and local URL for each page. In the new style there is no title. With no topic titles a CHM contents and index will be empty.

   For the purposes of conversion the main thing is to get the relative page URLs and the page titles in order to create a CHM. Parsing the pages themselves seems to be the only way to do that, but it works.

   The new style includes several files that may contain the data but it's in an unidentifed format, appearing only in a few files in the "$INDEX" folder and the BTREE file. Does Microsoft store the titles in the mysterious database-type files with names like "$LEAVES"? That seems unlikely because similar files exist in the old HXS style, but the HXT file still lists titles. Do they parse the HTML file TITLE tags in their own code? That seems likely. Perhaps Microsoft decided it was safer to parse webpages directly for their titles, in case the HXT file missed an update.

   There are also odd inconsistencies in path formatting of the URLs in the HXT file. Sometimes "/html/page1.htm", or "html/page1.htm", or even "html\page1.htm". The latter format is what the CHM help compiler seems to prefer, so all URLs are now converted to that form, not leaving the issue to the passing whims of Microsofties.

Fixes and Improvements in November 2010 update
1) Bad link fix. Some HXS files use this odd style of link:

<MSHelp:link TABINDEX="0" xmlns:MSHelp="http://msdn.microsoft.com/mshelp" keywords="_win32_Up_Down_Controls_cpp">Up-Down Controls</MSHelp:link>

Sometimes it seems like the Microsofties must spend most of their time thinking up ways to either break things or make them more complex. Instead of using an A tag for links, there is a long, custom tag named "MSHelp:link". That link contains a "keyword" attribute. The value of KEYWORD is a unique string. Each webpage file, in turn, will usually then contain a META tag with NAME "MS-HAID", where the CONTENT attribute is a keyword. To fix the MSHelp links it's necessary to map out all of those connections and then replace all MSHelp:link tags with functional A tags. This process can be time consuming on very big HXS files. The bigger the file, the slower the process. Unfortunately, the HXS system is full of quirks and the MSHelp links are no exception. An MS-HAID ID is not always present. In those cases the file base name seems to usually work as the ID, but that inconsistency in the "MSHelp" tags design means that all webpage files must be processed, even though many do not contain MSHelp links. Nevertheless, the operations to process the MSHelp links have been highly optimized to make them as fast as possible. Most files will still convert in seconds; the rare 30+ MB HXS file -- with perhaps 50,000 links in 10,000 pages -- may require something like 5 minutes for conversion.

2) Fix for superfluous and annoying script. In recent renditions of the Microsoft help files there is often a great deal of superfluous and problematic script used. Example: The document object topic page for Internet Explorer does not provide an option to show all properties and methods at once. One has to click the "Properties" link, then click a button to expand the list. Then to see the Methods list that link must be clicked...which then collapses the viewport...which requires another click on the expander button...and so on. The latest update of the converter script includes an option to remove all such nonsense during the conversion process. It removes all SCRIPT tags and all "display:none" CSS code, so that the full content of a topic page is always visible.

3) Fix for extra-long folder names that will crash the HTML Helkp compiler. The HTML Help compiler will freeze or crash if folder names are too long. Some HXS files have such folder names. Apparently it is not a problem for the HXS compiler. Due to the CHM compiler limitation, the folder tree needs to be checked when an HXS file is unpacked. After extracting the webpage files from an HXS, the script walks the folder tree and changes folder names if they are over 16 characters long. (Those changes must also be reflected in the HHP, HHC and HHK files. The script takes care of that.)

4) Favorites tab in all help files. A 4th change in the latest update is that CHM files produced from HXS files include a Favorites panel. Did you know that all CHM files can have a Favorites panel? It's especially handy with large files. A Favorites panel allows you to save links to often-visited topic pages. It's just like a Favorites menu in a browser. But the Favorites panel, inexplicably, is disabled by default! So most CHM files do not show it.

5) Option to customize the topic pages' appearance by adding your own CSS code. The converter script and converter HTA both provide the option to use custom CSS. If you provide a STYLE tag it will be inserted in each topic page, just above the </HEAD> tag. That means that your style preferences will override all other style settings except those that are inline. Custom style allows you to change font, font color, heading appearance, etc. Anything that can be done with CSS can be used to customize the finished CHM file.
